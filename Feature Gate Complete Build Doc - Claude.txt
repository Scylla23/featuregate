1. What You’re Building & Why It Matters
FeatureGate is a self-hosted feature flag platform that lets engineering teams control feature rollouts without redeploying code. It provides a management dashboard, a flag evaluation API, user segments with rule-based targeting, percentage rollouts, and a published Node.js SDK — essentially a lightweight, open-source alternative to LaunchDarkly.
Why This Project Is a Hiring Signal
Feature flag systems sit at the intersection of backend infrastructure, SDK design, and product engineering. Building one demonstrates that you understand progressive delivery, can design rule evaluation engines, and can ship developer tools that other engineers depend on. It’s the kind of project a Staff engineer would build, not a junior developer following a tutorial.
⚡ The Doubts This Kills:  "Can this person build platform/infrastructure tooling?" "Do they understand SDK contracts, caching, and real-time propagation?" "Can they design systems beyond CRUD?"
Your Differentiator vs. Existing Tools
	•	LaunchDarkly: Enterprise SaaS, starts at $10.50/seat/month. Powerful but expensive and closed-source.
	•	Unleash: Popular OSS (10K+ stars), but written in TypeScript with a complex setup and PostgreSQL dependency.
	•	Flipt: Lightweight Go binary, Git-native. Minimal UI and limited targeting.
	•	FeatureGate: Node.js-native (aligns with your stack), MongoDB-backed, simple Docker setup, first-class segments + targeting rules, and a beautiful React dashboard. Positioned for Node.js teams who want a LaunchDarkly-like experience without the cost.

2. Competitive Landscape — What to Learn From Each
LaunchDarkly — The Gold Standard
LaunchDarkly’s architecture reveals several design patterns you should adopt:
	•	Evaluation Order: Flag off check → Prerequisites → Individual targets → Targeting rules (top to bottom) → Default rule (fallthrough). This is the exact evaluation hierarchy you’ll implement.
	•	Segments: Reusable groups of users defined by rules or explicit member lists. A segment can be targeted by multiple flags, so you manage the audience in one place. Segments can include individual targets, custom rules (attribute-based), and even other segments.
	•	Targeting Rules: Each rule has conditions (attribute + operator + value) and a rollout (serve a specific variation or a percentage split). Rules are evaluated top-to-bottom; first match wins.
	•	Multivariate Flags: Flags can have multiple variations (not just true/false). Variations can be any JSON type: boolean, string, number, or object.
	•	Consistent Hashing: For percentage rollouts, LaunchDarkly hashes the user key to deterministically assign users to variations. Same user always gets the same variation.
	•	SDK Architecture: Server-side SDKs evaluate flags locally (in-memory cache of all flag data). Client-side SDKs call the server. Updates are streamed via SSE (Server-Sent Events) with ~200ms propagation.
Unleash — What to Adopt
	•	Docker-first deployment (docker-compose up and you’re running).
	•	Strategies pattern: activation strategies (gradual rollout, user IDs, IPs) that you attach to flags.
	•	Clear API docs and SDK conventions.
Flipt — What to Adopt
	•	Single-binary simplicity and lightweight footprint.
	•	Git-native flag storage (great for audit trails).
	•	Clean evaluation API design.

Component Responsibilities
	•	API Server (Node.js/Express): Flag CRUD, segment CRUD, evaluation endpoint, audit log, SSE broadcaster. This is the brain.
	•	MongoDB: Primary data store for flags, segments, environments, projects, and audit logs.
	•	Redis: Two roles — (1) Cache: store compiled flag evaluation data for <10ms reads. (2) Pub/Sub: broadcast flag change events to all connected SSE clients and SDK instances.
	•	React Dashboard: Management UI for creating/editing flags, segments, and viewing audit history.
	•	Node.js SDK: Published npm package that embeds in your application. Caches all flag data in memory, evaluates flags locally (zero network latency per check), and listens for updates via SSE.
Data Flow: Flag Evaluation
	•	SDK initializes: fetches all flag + segment data from GET /api/v1/flags/all and caches in memory.
	•	SDK opens SSE connection: GET /api/v1/stream. Server pushes flag.updated, segment.updated events.
	•	Application calls sdk.isEnabled('new-checkout', { userId: '123', plan: 'pro', country: 'US' }).
	•	SDK evaluates locally against cached flag data. No network call. Returns result in <1ms.
	•	When a flag is updated in the dashboard, API writes to MongoDB, invalidates Redis cache, publishes to Redis Pub/Sub channel.
	•	SSE broadcaster picks up the Pub/Sub message and pushes the updated flag data to all connected SDKs.
	•	SDK updates its in-memory cache. Next evaluation uses the new data. Total propagation: <500ms.

5. Flag Evaluation Engine (The Core Algorithm)
This is the most important piece of code in the entire project. The evaluation engine determines which variation a user gets. It runs both server-side (in the API for the /evaluate endpoint) and client-side (in the Node.js SDK for local evaluation).
🎯 Critical Principle:  The evaluation engine must be deterministic. Given the same flag configuration and user context, it must always return the same result. This is why percentage rollouts use consistent hashing, not Math.random().
Evaluation Order (exactly matches LaunchDarkly)
function evaluate(flag, context, segments) {
  // Step 1: Is the flag enabled?
  if (!flag.enabled) {
    return {
      value: flag.variations[flag.offVariation].value,
      variationIndex: flag.offVariation,
      reason: { kind: 'OFF' }
    };
  }
 
  // Step 2: Check individual targets (highest priority)
  for (const target of flag.targets || []) {
    if (target.values.includes(context.key)) {
      return {
        value: flag.variations[target.variation].value,
        variationIndex: target.variation,
        reason: { kind: 'TARGET_MATCH' }
      };
    }
  }
 
  // Step 3: Evaluate targeting rules (top to bottom, first match wins)
  for (let i = 0; i < (flag.rules || []).length; i++) {
    const rule = flag.rules[i];
    if (matchesRule(rule, context, segments)) {
      const result = resolveRollout(rule.rollout, context, flag);
      return {
        ...result,
        reason: { kind: 'RULE_MATCH', ruleIndex: i, ruleId: rule.id }
      };
    }
  }
 
  // Step 4: Fallthrough (default rule)
  const result = resolveRollout(flag.fallthrough, context, flag);
  return {
    ...result,
    reason: { kind: 'FALLTHROUGH' }
  };
}
Rule Matching
function matchesRule(rule, context, segments) {
  // ALL clauses must match (AND logic)
  return rule.clauses.every(clause => {
    const result = evaluateClause(clause, context, segments);
    return clause.negate ? !result : result;
  });
}
 
function evaluateClause(clause, context, segments) {
  // Special case: segment matching
  if (clause.attribute === 'segmentMatch') {
    return clause.values.some(segKey => 
      isUserInSegment(context, segments[segKey])
    );
  }
 
  const userValue = context[clause.attribute];
  if (userValue === undefined) return false;
 
  switch (clause.operator) {
    case 'in':        return clause.values.includes(userValue);
    case 'notIn':     return !clause.values.includes(userValue);
    case 'contains':  return clause.values.some(v => 
                        String(userValue).includes(v));
    case 'startsWith': return clause.values.some(v => 
                        String(userValue).startsWith(v));
    case 'endsWith':  return clause.values.some(v => 
                        String(userValue).endsWith(v));
    case 'matches':   return clause.values.some(v => 
                        new RegExp(v).test(String(userValue)));
    case 'greaterThan': return Number(userValue) > Number(clause.values[0]);
    case 'lessThan':  return Number(userValue) < Number(clause.values[0]);
    // semver operators omitted for brevity
    default:          return false;
  }
}
Segment Evaluation
function isUserInSegment(context, segment) {
  if (!segment) return false;
 
  // Excluded list takes highest priority
  if (segment.excluded.includes(context.key)) return false;
 
  // Included list takes second priority
  if (segment.included.includes(context.key)) return true;
 
  // Evaluate segment rules (OR logic: any rule match = in segment)
  return segment.rules.some(rule => {
    const clausesMatch = rule.clauses.every(clause => {
      const result = evaluateClause(clause, context, {});
      return clause.negate ? !result : result;
    });
 
    if (!clausesMatch) return false;
 
    // If weight is set, apply percentage check
    if (rule.weight !== undefined && rule.weight < 100000) {
      const bucketBy = rule.bucketBy || 'key';
      const bucket = hashUser(context[bucketBy], segment.key);
      return bucket < rule.weight;
    }
 
    return true;
  });
}
Consistent Hashing for Percentage Rollouts
This is what makes percentage rollouts deterministic. User "abc" always gets the same bucket, so they always see the same variation.
const murmurhash = require('murmurhash');
 
function hashUser(userKey, salt) {
  // Returns a value 0-99999 (maps to 0-100% in units of 0.001%)
  const hash = murmurhash.v3(String(userKey) + '.' + salt);
  return hash % 100000;
}
 
function resolveRollout(rollout, context, flag) {
  // Fixed variation
  if (rollout.variation !== undefined) {
    return {
      value: flag.variations[rollout.variation].value,
      variationIndex: rollout.variation
    };
  }
 
  // Percentage rollout
  const bucketBy = rollout.bucketBy || 'key';
  const bucket = hashUser(context[bucketBy], flag.key);
 
  let cumulative = 0;
  for (const weighted of rollout.variations) {
    cumulative += weighted.weight;
    if (bucket < cumulative) {
      return {
        value: flag.variations[weighted.variation].value,
        variationIndex: weighted.variation
      };
    }
  }
 
  // Fallback to last variation (shouldn't reach here if weights sum to 100000)
  const last = rollout.variations[rollout.variations.length - 1];
  return {
    value: flag.variations[last.variation].value,
    variationIndex: last.variation
  };
}


6. Segments & Targeting Rules — Deep Dive
This section covers the complete workflow for segments since this is your key requested feature.
6.1 How Segments Connect to Flags
Segments and flags are connected through the segmentMatch clause type. When a flag’s targeting rule has a clause with attribute: 'segmentMatch', the evaluation engine looks up the referenced segments and checks if the current user is a member.
// Example: Flag rule that targets a segment
{
  "rules": [
    {
      "id": "rule-1",
      "description": "Enable for beta testers",
      "clauses": [
        {
          "attribute": "segmentMatch",
          "operator": "in",
          "values": ["beta-testers"]   // segment key
        }
      ],
      "rollout": { "variation": 0 }   // serve 'true' variation
    },
    {
      "id": "rule-2",
      "description": "10% rollout for enterprise US",
      "clauses": [
        {
          "attribute": "segmentMatch",
          "operator": "in",
          "values": ["enterprise-us"]
        }
      ],
      "rollout": {
        "variations": [
          { "variation": 0, "weight": 10000 },  // 10% get true
          { "variation": 1, "weight": 90000 }   // 90% get false
        ]
      }
    }
  ]
}
6.2 Segment Use Cases
Use Case 1: Internal Team Testing
Create a segment "internal-team" with a rule: email endsWith '@yourcompany.com'. Use this across all feature flags to give your team early access to everything.
Use Case 2: Enterprise Tier Gating
Create a segment "enterprise-customers" with a rule: plan in ['enterprise']. When you launch a new enterprise-only feature, target this segment instead of hardcoding plan checks in every flag.
Use Case 3: Regional Rollout
Create segments "users-us", "users-eu", "users-apac" with country-based rules. Roll out a feature to US first (100%), then EU (50%), then APAC (10%) by targeting each segment with different rollout percentages.
Use Case 4: Beta Program with Opt-In
Create a segment "beta-opted-in" with: (a) explicit user IDs in the included list for VIP beta users, (b) a rule: betaOptIn in [true] for self-service opt-in, and (c) a rule: email endsWith '.edu' with weight 50000 for 50% of education users.
6.3 Segment Targeting in the Dashboard UI
The dashboard should provide a visual builder for segments. Here’s the UX flow:
	•	User navigates to Segments tab and clicks "Create Segment"
	•	Enters segment key, name, and description
	•	Adds included users (individual IDs, like LaunchDarkly’s individual targeting)
	•	Adds excluded users (override — always excluded regardless of rules)
	•	Builds rules visually: "IF [attribute] [operator] [values] AND ..." + optional % weight
	•	Saves segment. Dashboard shows which flags reference this segment.

The SDK is a separate npm package (@featuregate/node-sdk) that your users install in their applications. This is the public API contract — design it carefully.

const { FeatureGateClient } = require('@featuregate/node-sdk');
 
// Initialize (fetches all flags, opens SSE connection)
const client = new FeatureGateClient({
  sdkKey: 'sdk-prod-xxxx',
  baseUrl: 'https://your-featuregate-server.com',
  // Optional:
  pollingInterval: 30000,    // Fallback polling (if SSE disconnects)

await client.waitForInitialization();
 
// Evaluate a boolean flag
const enabled = client.isEnabled('new-checkout', {
  key: 'user-123',
  email: 'john@acme.com',
  plan: 'enterprise',
  country: 'US'
});
 
// Evaluate with a default value
const enabled2 = client.isEnabled('new-checkout', context, false);
 
// Get variation value (for multivariate flags)
const variant = client.variation('checkout-theme', context, 'control');
// Returns: 'control' | 'variant-a' | 'variant-b'
 
// Get evaluation detail (for debugging)
const detail = client.variationDetail('new-checkout', context, false);
// Returns: { value: true, variationIndex: 0, 
//            reason: { kind: 'RULE_MATCH', ruleId: '...' } }
 
// Get all flag values for a context (for frontend bootstrapping)
const allFlags = client.allFlagsState(context);
// Returns: { 'new-checkout': true, 'checkout-theme': 'variant-a', ... }
 
// Listen for flag changes
client.on('update', (flagKey) => {
  console.log(`Flag ${flagKey} was updated`);
});
 
// Cleanup
client.close();
SDK Internal Architecture
	•	FlagStore: In-memory Map<string, FlagConfig> holding all flag definitions.
	•	SegmentStore: In-memory Map<string, SegmentConfig> holding all segments.
	•	Evaluator: Pure function module (the same evaluation engine from Section 5). No side effects, fully testable.
	•	SSEClient: EventSource connection to /api/v1/sdk/stream. On flag.updated event, update FlagStore. On segment.updated, update SegmentStore.
	•	Poller: Fallback polling every N seconds if SSE disconnects. Calls GET /api/v1/sdk/flags.
	•	EventBuffer: Batches evaluation events and flushes to POST /api/v1/sdk/events every N seconds (for analytics).
SDK Error Handling
	•	If initialization fails (network error), the SDK should fall back to default values (not crash the host app).
	•	If SSE disconnects, switch to polling mode automatically. Reconnect SSE with exponential backoff.
	•	All public methods should be synchronous after initialization (they read from memory, no async needed).
	•	Emit events: 'ready', 'update', 'error' for the consuming application to listen to.

9. Real-Time Updates (SSE)
Server-Sent Events are the propagation mechanism. When a flag or segment changes, the update reaches all connected SDKs within milliseconds.
Why SSE Over WebSockets
	•	SSE is unidirectional (server → client), which is exactly what flag updates need. The SDK never sends data back through this channel.
	•	SSE auto-reconnects natively in browsers and most libraries. WebSockets require manual reconnection logic.
	•	SSE uses standard HTTP, so it works through proxies and load balancers without special configuration.
	•	This is a great interview talking point — it shows you know when NOT to use WebSockets.
Implementation
// Server: SSE endpoint
app.get('/api/v1/sdk/stream', authenticateSDK, (req, res) => {
  res.writeHead(200, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    'Connection': 'keep-alive',
  });
 
  // Send initial heartbeat
  res.write('event: heartbeat\ndata: {}\n\n');
 
  // Subscribe to Redis Pub/Sub
  const channel = `flag-updates:${req.environment.key}`;
  const subscriber = redis.duplicate();
  subscriber.subscribe(channel);
 
  subscriber.on('message', (ch, message) => {
    const event = JSON.parse(message);
    res.write(`event: ${event.type}\ndata: ${JSON.stringify(event.data)}\n\n`);
  });
 
  // Heartbeat every 30s to keep connection alive
  const heartbeat = setInterval(() => {
    res.write('event: heartbeat\ndata: {}\n\n');
  }, 30000);
 
  req.on('close', () => {
    subscriber.unsubscribe();
    subscriber.quit();
    clearInterval(heartbeat);
  });
});
 
// When a flag is updated, publish to Redis Pub/Sub
async function onFlagUpdated(flag, environmentKey) {
  const channel = `flag-updates:${environmentKey}`;
  await redis.publish(channel, JSON.stringify({
    type: 'flag.updated',
    data: { key: flag.key, flag: serializeFlag(flag) }
  }));
}

13.3 Cloud Infrastructure (Terraform)
You’ll provision all cloud resources with Terraform, version-controlled in the infra/ directory. Below are the two recommended paths — pick the cloud you’re more comfortable with.
Option A: AWS (ECS Fargate + ECR)
This is the recommended path. ECS Fargate is serverless containers — no EC2 instances to manage. You pay per vCPU-second.
Resources Terraform Creates
	•	VPC: Custom VPC with 2 public subnets (for ALB) and 2 private subnets (for ECS tasks).
	•	ECR: Two repositories — featuregate-server and featuregate-dashboard. Images pushed by CI/CD.
	•	ECS Cluster + Services: One Fargate service per container (server: 0.25 vCPU / 512MB, dashboard: 0.25 vCPU / 256MB). Auto-scaling based on CPU utilization.
	•	ALB (Application Load Balancer): Routes /api/* to the server container, /* to the dashboard container. TLS termination with ACM certificate. Health check paths configured.
	•	ElastiCache Redis: Single-node t3.micro (free tier eligible). Private subnet, security group allowing access only from ECS tasks.
	•	MongoDB Atlas: Free M0 cluster (512MB). Connected via VPC peering or IP allowlist. Alternatively, use AWS DocumentDB for a fully-AWS stack.
	•	CloudWatch: Log groups for both containers. Dashboard for container metrics (CPU, memory, request count).
	•	Route 53: Custom domain (featuregate.yourdomain.com) pointing to ALB.
	•	Secrets Manager: Store MONGO_URI, REDIS_URL, SDK secrets. Injected into ECS task definitions as environment variables.

Option B: GCP (Cloud Run + Artifact Registry)
Cloud Run is even simpler than ECS Fargate — it scales to zero, so you pay nothing when there’s no traffic. Great for a portfolio project.
Resources Terraform Creates
	•	Artifact Registry: Docker repo in your GCP project. Images pushed by CI/CD.
	•	Cloud Run Services: One per container. Auto-scales 0–3 instances. Custom domain mapping with managed TLS.
	•	Memorystore (Redis): Basic tier, 1GB. Connected via VPC connector.
	•	MongoDB Atlas: Free M0 cluster with GCP network peering.
	•	Cloud Logging: Automatic for Cloud Run — no config needed.
	•	Secret Manager: MONGO_URI, SDK keys. Mounted as env vars in Cloud Run.
Cloud Run Terraform (Simplified)
# infra/modules/cloudrun/main.tf
resource "google_cloud_run_v2_service" "server" {
  name     = "featuregate-server"
  location = var.region
 
  template {
    containers {
      image = var.server_image
      ports { container_port = 3000 }
      env { name = "NODE_ENV", value = var.environment }
      env { name = "REDIS_URL", value = var.redis_url }
      env {
        name = "MONGO_URI"
        value_source {
          secret_key_ref {
            secret  = google_secret_manager_secret.mongo_uri.id
            version = "latest"
          }
        }
      }
      startup_probe {
        http_get { path = "/healthz" }
        initial_delay_seconds = 10
      }
      liveness_probe {
        http_get { path = "/healthz" }
        period_seconds = 30
      }
      resources {
        limits = { cpu = "1", memory = "512Mi" }
      }
    }
    scaling {
      min_instance_count = 0     # Scale to zero!
      max_instance_count = 3
    }
    vpc_access {
      connector = google_vpc_access_connector.default.id
    }
  }
}

13.4 Health Check Endpoints
These are required for ALB/Cloud Run health checks and for your CI pipeline to verify deployments.
// packages/server/src/routes/health.ts
router.get('/healthz', (req, res) => {
  // Liveness: is the process running?
  res.status(200).json({ status: 'ok', uptime: process.uptime() });
});
 
router.get('/readyz', async (req, res) => {
  // Readiness: can the service handle requests?
  try {
    await mongoose.connection.db.admin().ping();
    await redis.ping();
    res.status(200).json({ 
      status: 'ready',
      mongo: 'connected',
      redis: 'connected' 
    });
  } catch (err) {
    res.status(503).json({ 
      status: 'not ready', 
      error: err.message 
    });
  }
});
13.5 Environment Strategy
Environment     Trigger                  Database         Cost
───────────────────────────────────────────────────────────────────
Local           docker-compose up        Local Mongo      Free
Staging         Push to main             Atlas free M0    ~$5/mo (Redis)
Production      Git tag (v*)             Atlas M0 / M10   ~$15/mo total
For a portfolio project, staging and production can share the same Atlas M0 free cluster with different database names. The key is that the deployment pipeline treats them as separate environments with separate Terraform state files.
13.6 Cost Breakdown (Portfolio-Friendly)
	•	MongoDB Atlas M0: Free (512MB, shared cluster). Plenty for a portfolio project.
	•	AWS ElastiCache t3.micro: ~$12/month (or use GCP Memorystore basic: ~$9/month).
	•	ECS Fargate (0.25 vCPU, 512MB): ~$9/month running 24/7 (or use Cloud Run scale-to-zero: ~$0-3/month).
	•	ECR/Artifact Registry: Free tier covers storage. ~$0.
	•	ALB: ~$16/month (or Cloud Run includes load balancing for free).
	•	Total: ~$15–40/month on AWS, or ~$10–15/month on GCP (Cloud Run). For a project that demonstrates cloud-native deployment skills, this is worth every rupee.
💡 Pro Tip:  Use GCP Cloud Run if you want to minimize costs (scale-to-zero = near-free). Use AWS ECS Fargate if you want maximum resume signal (AWS is still the most-requested cloud platform in US startup job postings). Either way, the Terraform code in your repo is what matters most.
